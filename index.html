Introduction to Git

Git is a distributed version control system(DVCS). Version control, also known as source control, is the practice of tracking and managing changes to software code. Version control systems are software tools that help software teams manage changes to source code over time.Distributed version control is a form of version control in which the complete codebase, including its full history, is mirrored on every developer's computer.
Git does not rely on any central server and that is why you can perform many operations when you are offline. You can commit changes, create branches, view logs(we will cover those terms shortly) and perform other operations while offline. You require a network connection only to publish your changes and take the latest changes from the remote repository(also to be covered shortly).
It uses SHA1 to name and identify objects within its database. Every file and commit is check-summed and retrieved by its checksum at the time of checkout.

Importance of Git

The use of git is very important and the following points help to illustrate why:

  1. Backup: Git helps you to maintain the backup of your source code.
  2. Collaboration: Git helps you to collaboratively work with N number of developers.
  3. Easy code deployment: Git helps you to deploy your source code on server with just one command.
  4. History: Git helps you to view/rebase the project history. You can find out who changed particular file and which lines are changed.
  5. Branching and merging: Branching helps to create and integrate new features in existing project without affecting the working code. (Only if you follow proper git flow).
  6. Cloud Storage: Git is supported by Github which is cloud storage for your code base. Github supports both public and private repositories.
  7. Speed: Git is faster than other version controlling systems.
  8. Time travelling: You can revisit a commit that you did sometime ago and correct it using interactive rebase.

Git and Github

There is a common misconception that git and github are one and the same thing. It is Important that we clear this confusion before going ahead with the documentation.
Git and github perform different functions and serve different purposes. The two are not even owned by the same company. Git is an open source tool maintained by the Linux Foundation. Github, by contrast is owned by microsoft.
Git is a free, open-source software distributed version control system designed to manage all source code history. It can keep a history of commits, can reverse changes, and lets developers share code. Each developer must have git installed on his or her local device to collaborate.
GitHub, on the other hand, is a web-based hosting service for git repositories. It offers all of Git’s distributed version control system source code management and has some additional features. This includes collaboration functionality like project management, support ticket management, and bug tracking. With GitHub, developers can share their repositories, access other developers’ repositories, and store remote copies of repositories to serve as backups.

DVCS Terminologies

Local Repository

Git provides a private workspace as a working copy. Developers make changes in their private workspace and after commit, these changes become part of the repository.

Working directory and staging area/index

The working directory is the place where the files are checked out.
Whenever you commit an operation, git looks for the files in the staging area. Only those files present in the staging area are considered for commit and not all the modified files.
Thus this is the basic git workflow:

  • Step 1: modify a file from the working directory
  • Step 2: you add these files to the staging area
  • Step 2: you perform a commit operation that moves the files from the staging area. After push operation, it stores the changes permanently to the git repository

Blobs

Blob - Binary Large Object. Each version of a file is represented by a blob. A blob holds the file data but does not contain any metadata about the file. It is a binary file and in git database.

Trees

An object which represents a directory. Holds blobs as well as other sub-directories.

Commits

A commit holds the current state of the repository. A commit is also named by SHA1 hash.

Branches

Branches are used to create another line of development. By default, git has a master branch. Usually a branch is created to work on a new feature. Once a feature is completed, it is merged back with the master branch and we delete the branch.

Tags

Tag asssigns a meaningful name with a specific version in the repository. Tags are similar to branches, but the difference is that tags are immutable.
Once a tag is created for a particular commit, even if you create a new commit, it will not be updated. Usually, developers create tags for product releases.

Clone

Creates an instance of the repository.

Pull

Pull operations copies the changes from a remote repository instance to a local one.

Push

Push operation copies changes from a local repository to a remote one. This is used to store the changes permanently into the git repository.

HEAD

HEAD is a pointer, which always points to the latest commit in the branch. Whenever you make a commit, HEAD is updated with the latest commit.
The heads of the branches are stored in .git/refs/heads directory.

Revision

Revision represents the version of the source code. Revisions in git are represented by commits. Commits are identified by SHA1 secure hashes.

URL

Represents the location of the git repository. Git url is stored in config file.

Basic Git Commands
  1. git init: Creates a new local git repository.

    git init

    Alternatively, you can create a new repository within a new directory by specifying the project name.

    git init [project name]

  2. git clone: Used to copy a repository.
    From a remote server.

    git clone username@host:/path/to/repository

    From a local repository

    git clone /path/to/repository

  3. git commit: Creates a snapshot of the changes and saves it to the git directory.

    git commit -m "message to go with commit"

  4. git config: Used to set user specific configuration values e.g email, username etc

    git config --global user.email youremail@sm.com

    git config --global user.name 'username'

    The global flag tells git that you are going to use that email for all local repositories. To use different emails for different repositories, use the --local flag.
  5. git status: displays the list of changed files together with the files that are yet to be staged/commited.

    git status

  6. git push: sends local commits to whatever branch we specify of the remote repository.

    git push origin [branch name]

  7. git checkout: Creates branches and helps you navigate between them. To create a branch and automatically switch to it:

    git checkout -b [branch name]

    To switch from one branch to another

    git checkout [branch name]

  8. git remote: Lets you view all remote repositories. To list all connections, along with url.

    git remote -v

    To connect the local repository to a remote server.

    git remote add origin [host of remoteURL]

    Delete a connection to a specified remote repository.

    git remote rm [name of the repository]

  9. git branch: List, create or delete branches. To list all branches in the repository:

    git branch

    To delete a branch:

    git branch -d [branch name]

  10. git pull: merges all the changes in the remote repository to the local working directory:

    git pull origin [branch name]

  11. git merge: merges a branch into the active one:

    git merge [branch name]

  12. git diff: Lists down conflicts. To view conflicts against the base file

    git diff --base [file name]

    To view conflicts between branches before merging them.

    git diff [source branch] [target branch]

    List down all present conflicts.

    git diff

  13. git tag: marks specific commits. Developers use it to mark release points eg v1.0 and v2.0

    git tag [insert-commitID-here]

  14. git log: used to see the repository's history by listing certain commit's details.

    git log

  15. git reset: resets the index and the working directory to the last git commit's state.

    git reset --hard HEAD

  16. git rm: removes files from the index and the working directory.

    git rm [filename]

  17. git stash: temporarily saves changes that are not ready to be committed. So you can go back to the project later.

    git stash

  18. git show: used to view information about every git object

    git show

  19. git fetch: allows users to fetch all objects from the remote repository that do not currently reside in the local working directory.

    git fetch origin

  20. git ls-tree: allows you to view a tree object along with the name, the mode of each item and the blob SHA1 value. Say you want to view the HEAD:

    git ls-tree HEAD

  21. git cat-file: view the type and size information of a repository object. Use the -p option along with the object's SHA1 value to view information of a specific object

    git cat-file -p [SHA1 value]

  22. git grep: lets users search through committed trees, working directory and staging area for specific phrases and words. Example: to search for the word "aloha"

    git grep 'aloha'

  23. gitk: shows the graphical interface for a local repository

    gitk

  24. git instaweb: allows you to browse your local repository in the git-web interface. Example:

    git instaweb --httpd=webrick

  25. git gc: will clean unecessary files and optimize local repository.

    git gc

  26. git archive: lets users create a zip/tar file containing the constituents of a single repository tree. Example

    git archive --format=tar master

  27. git prune: deletes objects that do not have any incoming pointers.

    git prune

  28. git fsck: performs an integrity check of the git sile system and indentifies any corrupted objects.

    git fsck

  29. git rebase: used to apply certain changes from one branch to another. Example:

    git rebase master
